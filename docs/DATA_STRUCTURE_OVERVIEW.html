Data-Oriented View

JYEngine 현재 데이터 구조도

이 문서는 현재 엔진을 데이터 관점에서만 본 구조도입니다. 핵심은 Scene이 CPU-side source pool을 들고, RenderServer가 그것을 frame용 render item으로 조합하며, RenderDB가 GPU mirror/cache를 유지하고, Renderer가 그 결과만 소비하는 흐름입니다.

Source Data JScene + pool instances
Pool Logic JPool<Handle, Data>
Frame Build JRenderServer::BuildFrameDesc()
GPU Mirror JRenderDB

상위 데이터 흐름

1. Scene Source Data

JScene 현재 씬의 CPU-side truth source
→
Entity / Transform / Camera / Light / RenderObject Pools handle 기반 pool 인스턴스

2. Frame Assembly

JRenderServer scene를 읽고 visible render item 조합
→
FrameDesc camera, lights, opaque items, transparent items

3. GPU Mirror / Cache

JRenderDB material/camera/mesh GPU-side resource mirror
→
MaterialResource / CameraResource / MeshResource draw에서 바로 쓰는 렌더 데이터

4. Final Consumption

JRenderer FrameDesc와 RenderDB만 소비
→
JGraphicResource draw 1회용 binding 묶음
→
JCommandQueue 실제 draw command 기록
현재 구조에서 중요한 점은 Renderer가 Scene pool을 직접 뒤지지 않는다는 점입니다. Scene의 raw data는 RenderServer에서 정리되고, Renderer는 FrameDesc와 RenderDB만 봅니다.

프레임 데이터 흐름

1

Scene에 데이터 저장

JScene가 entity, transform, camera, render object를 pool에 저장합니다.

2

RenderServer가 source data 참조

현재 scene의 pool을 읽고 유효한 render object와 primary camera를 선택합니다.

3

FrameDesc 생성

camera, lights, opaqueDrawItems, transparentDrawItems를 조합합니다.

4

RenderDB sync

material/camera는 dirty 기반으로, mesh는 cache 기반으로 GPU-side mirror를 유지합니다.

5

Renderer 소비

Renderer는 draw item의 material/mesh 정보를 가지고 RenderDB에서 resource를 찾아 draw합니다.

핵심 데이터 철학

  • JScene은 CPU-side source data owner입니다.
  • JPool은 공용 저장/유효성 로직입니다.
  • JRenderServer는 source data를 frame-friendly render data로 압축합니다.
  • JRenderDB는 GPU mirror/cache만 담당합니다.
  • JRenderer는 pool을 직접 탐색하지 않고 정규화된 frame data만 소비합니다.
한 줄로 요약하면:
Scene holds data → RenderServer assembles data → RenderDB mirrors data → Renderer consumes data
