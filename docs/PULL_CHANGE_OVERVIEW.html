Pulled Change Overview

Move scene rendering through JRenderer and RenderDB

최신 pull의 핵심은 JScenePanel이 직접 렌더링 자원을 만들고 그리던 구조에서 벗어나, JMaterialFactory, JRenderServer, JRenderDB, JRenderer를 거쳐 씬 렌더링을 수행하도록 책임을 재배치한 것입니다.

69716f9 최신 핵심 커밋
18 files 직접 수정된 파일
4 files 새로 추가된 파일
Scene → Renderer 렌더 경로 이동

구조도

Editor / Client Layer
JScenePanel 카메라 입력, 메시 로드, material 생성, frameDesc 작성
Engine Services
JMaterialFactory Shader / Pipeline / CBV / Texture 생성 helper
→
JRenderServer Material / Camera 등록, dirty 관리, Sync
→
JRenderDB MaterialResource / CameraResource / MeshResource 캐시
→
JRenderer FrameDesc 기반 draw loop 실행
Render Backend
JGraphicResource Draw 1회용 shader binding 묶음
→
JCommandQueue Pipeline, CBV, SRV, draw 호출 기록
→
DX12 실제 command list / swap chain 출력
이전 구조는 ScenePanel이 shader, pipeline, vertex/index buffer, graphic resource까지 직접 소유했습니다. 현재 구조는 씬 패널이 authoring/조립 역할만 맡고, 렌더링 실행은 엔진 쪽 서비스로 이동했습니다.

렌더 흐름

1

Material 생성

JMaterialFactory가 shader와 pipeline을 묶어 JMaterial을 만듭니다.

2

ScenePanel 등록

JScenePanel이 material, camera, mesh를 준비하고 JRenderServer에 등록합니다.

3

Sync

JRenderServer::Sync()가 dirty material / camera를 JRenderDB로 반영합니다.

4

FrameDesc 작성

JScenePanel이 cameraID, renderTarget, viewport, drawItems를 채워 JRenderer로 넘깁니다.

5

Draw 실행

JRenderer가 material resource를 읽고 JGraphicResource를 만들어 draw를 실행합니다.

추가된 핵심 타입

src/engine/JRenderer.*

렌더링 실행 계층. FrameDesc와 DrawItem을 입력받아 draw loop를 수행합니다.

src/engine/JMaterialFactory.*

material authoring helper. shader/pipeline 생성과 CBV/texture 설정을 묶습니다.

src/engine/JRenderDB.*

material뿐 아니라 camera, mesh까지 렌더러용 캐시를 보관합니다.

src/engine/JRenderServer.*

source object와 RenderDB 사이의 동기화 계층입니다.

바뀐 책임

  • JScenePanel: 직접 draw 호출하는 구조에서, render request 작성 쪽으로 축소
  • JMaterial: 이제 shader와 pipeline도 소유
  • JRenderDB: camera per-frame data와 mesh GPU resource까지 관리
  • JEngine: JRenderer, JMaterialFactory를 새로 소유
  • JMeshResource: 단순 view 묶음에서 실제 vertex/index buffer 포인터 소유 구조로 확장
Scene Simplified Renderer Introduced Camera Sync Added Material Factory Added RenderDB Expanded

현재 읽는 포인트

  • 좋아진 점: 패널 코드가 가벼워지고, 렌더링 실행 책임이 엔진 서비스로 모였습니다.
  • 구조 의미: 향후 Actor/Component, 여러 panel, render graph로 확장하기 쉬운 방향입니다.
  • 남은 숙제: draw마다 JGraphicResource를 매번 재생성하는 부분과 mesh cache의 소유권 모델은 더 다듬을 여지가 있습니다.
