Scene 이후 DOD Render Flow

목표는 Scene 이후 렌더링 경로를 단방향 데이터 파이프라인으로 고정하는 것이다. Scene은 CPU 원본, Snapshot은 프레임 입력, RenderDB는 GPU 리소스 테이블, DrawItem은 정렬 가능한 실행 단위가 된다.

최종 데이터 흐름

Scene Pools

  • CPU 원본 데이터
  • TransformPool SoA
  • RenderObjectPool handle/index
  • GPU 포인터 없음
→

Frame Snapshot

  • 렌더에 필요한 CPU 계산 결과
  • world matrix
  • light position/color
  • render object references
→

RenderDB

  • GPU resource owner
  • resource table index 발급
  • mesh/material/transform/camera 관리
  • 추후 big buffer pack
→

DrawItem List

  • 정렬 가능한 draw 실행 단위
  • pointer 대신 index/offset
  • sortKey 보유
  • Renderer가 순차 소비
Current Risk

현재 애매한 형태

struct JDrawItem
{
    JEntityHandle entity;
    JRenderObjectHandle renderObject;
    uint32 materialID;
    const JMesh* mesh;

    const JMeshResource* meshResource;
    const JMaterialResource* materialResource;
    const JTransformResource* transformResource;
    bool transparent;
};
  • DrawItem 배열 자체는 연속적이다.
  • 하지만 resource pointer를 따라가면 랜덤 접근이 생긴다.
  • lifetime도 RenderDB 소유권과 섞여 보인다.
  • big buffer로 가면 포인터보다 offset/index가 맞다.
Target

목표 DrawItem

struct JDrawItem
{
    uint64 sortKey = 0;

    uint32 meshResourceIndex = INVALID_INDEX;
    uint32 materialResourceIndex = INVALID_INDEX;
    uint32 transformResourceIndex = INVALID_INDEX;

    uint32 perObjectOffset = 0;
    uint32 perMaterialOffset = 0;

    uint32 indexStart = 0;
    uint32 indexCount = 0;
    int32 vertexBase = 0;

    uint8 passMask = 0;
};
  • Scene pointer 없음.
  • GPU resource pointer 없음.
  • 정렬과 batching 기준을 가진다.
  • Renderer hot path에서 순차 처리하기 좋다.

책임 분리

Scene
CPU 원본 상태만 보관한다. 에디터 입력과 게임 로직은 여기 데이터를 바꾼다.
Snapshot
한 프레임에 필요한 렌더 입력을 고정한다. GPU 리소스는 들지 않는다.
RenderDB
GPU 리소스와 dense resource table을 소유한다. Scene은 보지 않는다.
Renderer
DrawItem list를 순차 소비한다. Scene query와 unordered lookup을 피한다.

RenderObject 최종 방향

렌더 경로에서 entity를 통해 transform을 다시 찾지 않도록 render object가 transform handle을 직접 가진다.

현재
RenderObject -> entity -> transform lookup
목표
RenderObject -> transformHandle

Sort Key 방향

Opaque는 state change 감소가 우선이고, Transparent는 거리 정렬이 우선이다.

Opaque
pipeline -> material -> mesh
Transparent
distance back-to-front

권장 리팩토링 순서

RenderObjectData에 transform handle 추가

Scene 이후 렌더 경로에서 entity lookup을 줄이는 가장 작은 DOD 개선이다.

JFrameSnapshot에서 DrawItem 분리

Snapshot은 CPU 렌더 입력, DrawList는 렌더 실행 목록으로 책임을 나눈다.

JDrawItem을 pointer-free index 기반으로 전환

RenderDB resource pointer 대신 dense resource index와 sortKey를 보관한다.

FrameDesc vector copy 제거

DrawItem list를 복사하지 말고 view/reference로 Renderer에 전달한다.

Transform constant buffer부터 big buffer화

DrawItem의 transform index가 per-object offset으로 자연스럽게 이어진다.
