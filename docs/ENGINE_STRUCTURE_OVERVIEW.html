Current Engine Overview

JYEngine 전체 데이터 구조와 렌더 흐름

현재 기준의 JYEngine은 high-level object layer, render sync/cache layer, render execution layer, DX12 backend로 나뉩니다. 핵심 철학은 사용자 객체와 렌더러용 파생 데이터 분리, 그리고 ScenePanel은 조립하고 Renderer가 실행한다입니다.

Object Layer JObject, JMaterial, JCameraComponent
Sync Layer JRenderServer, JRenderDB
Draw Layer JRenderer, JGraphicResource
Backend JCommandQueue, JRenderContext, DX12

전체 레이어 구조

Client / Editor

JScenePanel 카메라 입력, 메시 로드, material 조립, frameDesc 작성

High-Level Objects

JObject ID, 이름, PendingKill 플래그를 가지는 base object
→
JMaterial shader, pipeline, constant buffer, texture authoring object
JCameraComponent position, yaw, pitch, view/projection 계산
JMesh CPU-side mesh 데이터 컨테이너

Sync / Derived Data

JMaterialFactory shader/pipeline/CBV/texture 생성 helper
→
JRenderServer material/camera 등록, dirty tracking, Sync orchestration
→
JRenderDB MaterialResource, CameraResource, MeshResource 캐시

Render Execution

JRenderer FrameDesc를 받아 draw loop 실행
→
JGraphicResource draw 1회용 binding 묶음
→
JCommandQueue pipeline, root signature, CBV/SRV, draw command 기록
→
DX12 command list, swap chain, RTV, descriptor heap 실행

엔진 루트 서비스

JEngine

현재 엔진 루트는 아래 서비스를 소유합니다.

JDevice, JCommandQueue, JSwapChain, JRenderContext, JRenderServer, JRenderer, JMaterialFactory

JRenderContext

GPU resource 생성 계층입니다.

CreateShader, CreatePipeline, CreateConstantBuffer, CreateSolidColorTexture를 담당합니다.

JSwapChain + JCommandQueue

프레임 시작/종료, render target 전환, 실제 draw command 제출을 담당합니다.

현재 구조의 중요한 포인트는 ScenePanel이 GPU API를 직접 건드리지 않고, 엔진 서비스 계층을 통해서만 렌더링을 요청한다는 점입니다.

핵심 데이터 구조

JMaterial

  • JObject 상속
  • JShader*, JPipeline* 소유
  • 이름/해시 기반 CBV 목록
  • 이름/해시 기반 texture 목록
  • dirty 플래그 보유

JMaterialResource

  • RenderDB 내부 파생 데이터
  • materialID 기준 관리
  • shader/pipeline 캐시
  • CBV/texture binding entry 배열

JGraphicResource

  • 실제 draw 직전에 생성
  • root parameter index 포함
  • CBV binding 배열
  • texture binding 배열

JRenderDB::CameraResource

  • cameraID
  • viewProjection
  • perFrameBuffer

JMeshResource

  • GPU vertex buffer 포인터 목록
  • GPU index buffer 포인터
  • VBV / IBV
  • indexSize

JCameraComponent

  • position
  • yaw, pitch
  • moveSpeed, rotateSpeed
  • GetViewMatrix(), GetProjectionMatrix()

현재 샘플 씬 기준 소유 관계

JScenePanel가 직접 들고 있는 것

  • material, planeMaterial
  • mesh, planeMesh
  • meshResource, planeMeshResource
  • perFrameBuffer, materialBuffer, materialTexture
  • camera

RenderServer / RenderDB가 파생해서 들고 있는 것

  • material별 JMaterialResource
  • camera별 CameraResource
  • mesh별 JMeshResource 캐시

draw 시점에만 생기는 것

  • JRenderer 내부의 local JGraphicResource
  • FrameDesc.drawItems

프레임 단위 흐름도

1

Init 단계

JScenePanel이 JMaterialFactory로 material을 만들고, camera와 mesh를 준비합니다.

2

Register 단계

RegisterMaterial(), RegisterCamera(), CreateOrUpdateMeshResource()로 source object를 render 계층에 연결합니다.

3

Update 단계

카메라 입력을 처리하고, dirty가 생긴 material/camera를 JRenderServer에 마킹합니다.

4

Sync 단계

JRenderServer::Sync()가 source data를 읽어 JRenderDB의 material/camera resource를 갱신합니다.

5

FrameDesc 작성

cameraID, renderTarget, viewport, drawItems를 채워 renderer에 넘깁니다.

6

Render 단계

JRenderer가 material resource를 조회하고 JGraphicResource를 빌드한 뒤 draw를 실행합니다.

실제 데이터 이동

Material 경로

JMaterial → JRenderServer::MarkMaterialDirty() → JRenderDB::SyncMaterial() → JMaterialResource → JGraphicResource

Camera 경로

JCameraComponent → JRenderServer::MarkCameraDirty() → JRenderDB::SyncCamera() → CameraResource.viewProjection → perFrameBuffer

Mesh 경로

JMesh → JRenderDB::CreateOrUpdateMeshResource() → GPU vertex/index buffer → DrawItem.meshResource

Draw 경로

FrameDesc.drawItems → JRenderer → SetPipeline() → SetGraphicResources() → BindVertexBuffer() → DrawIndexed()

현재 구조를 한 줄로 요약하면

JScenePanel은 사용자 레벨 오브젝트를 만들고 상태를 바꾸는 쪽에 머물고, JRenderServer와 JRenderDB가 렌더러용 파생 데이터를 유지하며, JRenderer가 프레임 기술자 FrameDesc를 받아 실제 draw를 수행하는 구조입니다.

이 구조는 나중에 JActor / JComponent, 더 많은 panel, render graph, frame capture, AI-friendly JSON export로 확장하기 좋은 중간 형태입니다.
